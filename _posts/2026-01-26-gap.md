### 개요

Node.js + Express로 만든 간단한 렌더링 서비스입니다.
`POST /render`에 입력을 보내면, 서버가 고정된 템플릿 `<%= input %>`에 값을 채워 반환합니다.
플래그는 서버의 `/flag.txt`에 저장되어 있습니다.

처음 보면 안전해 보입니다. 템플릿이 `<%= input %>`으로 고정되어 있으니, 템플릿 자체를 바꾸는 SSTI는 통하지 않습니다.
하지만 핵심은 **"무엇을 어디에 넘기는가"** 입니다.

---

### 취약점 분석

lodash의 `_.template()`은 두 개의 인자를 받습니다.

```js
const compiled = _.template(templateString, options);
compiled(data);  // 렌더링
```

첫 번째는 템플릿 문자열, 두 번째는 **컴파일 옵션**입니다.
서버 코드는 `req.body`를 렌더링 데이터(`data`)로만 써야 하는데, 실수로 `options`에도 그대로 흘려보내고 있습니다.

```js
// 의도한 코드
const compiled = _.template("<%= input %>");
compiled(req.body);  // { input: "hello" }

// 실제 코드 (취약)
const compiled = _.template("<%= input %>", req.body);  // ← options 자리에 req.body가 들어감
compiled(req.body);
```

이 `options` 중에 `imports`라는 키가 있습니다.
`imports`는 템플릿 컴파일 시 **함수 스코프에 주입할 변수**들을 정의하는 옵션입니다.

lodash는 내부적으로 `imports`의 키를 컴파일된 함수의 **매개변수 이름**으로 사용합니다.
그런데 이 키 이름이 JavaScript 식별자라면, 다음처럼 기본값(default parameter)을 포함한 문자열을 키로 넣을 수 있습니다.

```
키: "a, b, input=<실행할 코드>"
```

lodash가 내부적으로 만드는 함수는 이렇게 됩니다.

```js
function(a, b, input = fs.readFileSync("/flag.txt", "utf8")) {
    // 템플릿 본문: return input;
}
```

`input`을 따로 전달하지 않으면 JavaScript는 **기본값(default value)을 평가**합니다.
기본값 자리에 파일 읽기 코드를 넣으면, `<%= input %>`이 `/flag.txt`의 내용을 그대로 출력합니다.

정리하면, 취약점은 두 가지가 맞물린 결과입니다.

**1. `req.body`가 `options` 자리에 새어나감:**
개발자가 렌더링 데이터와 컴파일 옵션을 구분하지 않고 같은 객체를 넘겼습니다.

**2. lodash `imports` 키가 함수 파라미터 이름으로 그대로 삽입됨:**
키 이름에 `=<코드>`를 붙이면 기본값 표현식이 되어, 임의 JavaScript가 실행됩니다.

### 요약

이 문제의 핵심은 **"데이터와 옵션의 경계가 무너진 것"** 입니다.

`req.body`가 렌더링 데이터임과 동시에 컴파일 옵션으로 흘러들어가는 순간, 공격자는 `imports` 옵션을 통해 템플릿 함수의 파라미터 정의 자체를 조작할 수 있게 됩니다.

템플릿 문자열(`<%= input %>`)은 건드리지 않아도 됩니다.
`input`이라는 변수가 **어떻게 정의되는지**를 바꾸는 것만으로 충분합니다.

lodash `_.template()`을 사용할 때, 사용자 입력이 두 번째 인자(`options`)에 절대 섞여들지 않도록 주의해야 합니다.
