# Netapp (BITS CTF 2026)

---

- **Name:** Netapp
- **Category:** Web
- **Description:** We think our internal services are protected because we use netapp 😘
- **Connection:** `https://netapp.bitskrieg.in`
- **Flag format:** `BITSCTF{...}`

---

### 개요

사이트에 접속하면 서비스 상태 대시보드가 보입니다.

```
Web Portal    — 🟢 ONLINE
Auth Gateway  — 🟢 ONLINE
flag-service  — 🟡 INTERNAL (VPN only)
```

플래그는 `flag-service`에 있는데, VPN 없이는 접근할 수 없다고 합니다.
VPN 없이 내부 서비스에 도달하는 것이 목표입니다.

### 소스 분석

#### .git 디렉터리 발견

`ffuf`로 디렉터리를 브루트포싱해보면 `.git`이 노출되어 있습니다.

```bash

```

```bash

```

덤프된 파일 중 의미 있는 것들이 보입니다.

```
flag-service/
  ├── aws-security-group.tf
  └── bitsctf-2026-vpn-only.tf
```

Terraform 인프라 코드입니다.
여기서 서버 구성을 전부 읽을 수 있습니다.

#### aws-security-group.tf — EC2 IP와 방화벽 규칙

```hcl
# EC2 IP: 3.208.18.209

resource "aws_security_group" "cf_only_web" {
  ingress {
    from_port   = 80
    cidr_blocks = [
      "173.245.48.0/20",
      "103.21.244.0/22",
      # ... Cloudflare IP 대역만 허용
    ]
  }
}
```

EC2의 실제 IP(`3.208.18.209`)를 알았고,
AWS 보안 그룹이 **Cloudflare IP 대역에서 오는 요청만** 허용한다는 것도 확인됩니다.

직접 `http://3.208.18.209`에 접근하면 타임아웃입니다.
일반 사용자 IP는 보안 그룹에서 막힙니다.

#### bitsctf-2026-vpn-only.tf — Cloudflare 방화벽 규칙

```hcl
resource "cloudflare_filter" "vpn_access_filter" {
  body = "(http.host eq \"bitsctf-2026.hvijay.dev\") and (ip.src ne 0.0.0.0)"
}
resource "cloudflare_firewall_rule" "vpn_access_rule" {
  action = "block"
}
```

Cloudflare 방화벽은 `bitsctf-2026.hvijay.dev` 도메인으로 오는 요청을 전부 차단합니다.
`https://bitsctf-2026.hvijay.dev`에 직접 접근하면 403입니다.

### 인프라 구조 정리

두 파일을 합치면 보호 레이어가 두 겹인 구조가 됩니다.

```
외부 요청
    │
    ▼
Cloudflare 방화벽
  host = "bitsctf-2026.hvijay.dev" → 차단 ❌
    │
    ▼
AWS EC2 (3.208.18.209)
  보안 그룹: Cloudflare IP만 허용
  일반 IP → 타임아웃 ❌
```

1. **Cloudflare 방화벽** — 해당 도메인으로 오는 외부 요청을 차단
2. **AWS 보안 그룹** — Cloudflare IP 대역에서 온 요청만 통과

두 제약을 동시에 만족해야 플래그에 도달할 수 있습니다.

---

### 우회 시도들

이것저것 시도해봅니다.

- `https://bitsctf-2026.hvijay.dev` 직접 접근 → **403** (Cloudflare 방화벽)
- `http://3.208.18.209` 직접 접근 → **타임아웃** (AWS 보안 그룹)
- `X-Forwarded-For: 0.0.0.0` 헤더 추가 → **403** (Cloudflare가 무시함)
- Cloudflare Worker에서 EC2 IP 직접 fetch → **Error 1003** (CF가 raw IP fetch 차단)
- Cloudflare Worker에서 도메인 fetch → **403** (방화벽 규칙 여전히 적용)

Worker에서 IP를 직접 fetch하면 Cloudflare 자체가 막습니다.
도메인으로 fetch하면 방화벽 규칙에 걸립니다.

**IP도 안 되고 도메인도 안 된다면, IP를 도메인처럼 쓰면?**

### 취약점 분석

#### nip.io — IP를 도메인 이름으로

`nip.io`는 와일드카드 DNS 서비스입니다.
`3.208.18.209.nip.io`는 `3.208.18.209`로 그대로 해석됩니다.

이게 두 제약을 동시에 우회합니다.

**Cloudflare 방화벽 우회:**
방화벽 규칙의 조건은 `host = "bitsctf-2026.hvijay.dev"`입니다.
`3.208.18.209.nip.io`는 이 조건에 해당하지 않으므로 차단되지 않습니다.

**AWS 보안 그룹 통과:**
Cloudflare Worker는 Cloudflare 인프라에서 실행됩니다.
요청의 출발 IP가 Cloudflare IP 대역이므로 보안 그룹을 통과합니다.

```
[Cloudflare Workers]
    │
    │ fetch("http://3.208.18.209.nip.io/")
    ▼
[Cloudflare 방화벽]
  host = "3.208.18.209.nip.io"
  ≠ "bitsctf-2026.hvijay.dev"
  → 규칙 미적용 → 통과 ✅
    │
    ▼
[AWS EC2: 3.208.18.209]
  출발 IP = Cloudflare Worker IP
  → 보안 그룹 허용 → 통과 ✅
  → 플래그 반환
```

### 요약

이 문제의 핵심은 **두 방어 레이어의 조건이 서로 다르다는 점**을 이용하는 것입니다.

AWS 보안 그룹은 "Cloudflare IP에서 온 요청인가"를 봅니다.
Cloudflare 방화벽은 "특정 도메인으로 온 요청인가"를 봅니다.

두 조건을 동시에 만족하는 경로가 있습니다.
Cloudflare Worker에서 `nip.io` 도메인으로 요청을 보내면,
도메인 조건은 불일치로 방화벽을 통과하고,
Cloudflare IP 조건은 Worker 인프라 덕에 보안 그룹을 통과합니다.

노출된 `.git`에서 Terraform 코드를 읽어 인프라 구조를 파악한 것이 시작이었습니다.
인프라 코드가 공개되면 방어 로직 자체가 노출됩니다.
